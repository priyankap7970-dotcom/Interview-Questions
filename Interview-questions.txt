## 🔹 Project Architecture & Design
🔹 1. Explain your end-to-end Azure DevOps project architecture (from code commit to production deployment).
Our application was a microservices-based workload deployed on AKS. Developers pushed code to Azure Repos using feature branches. Once the PR was approved and merged into main, the YAML CI pipeline triggered automatically.
The CI pipeline built the application, ran unit tests, executed SonarQube quality checks, built a Docker image, and pushed it to Azure Container Registry. Only if all validations passed was the image published.
The CD pipeline then deployed the same immutable image across Dev, QA, UAT, and Prod using Helm charts. Infrastructure like AKS, ACR, VNet, and Key Vault was provisioned using Terraform.
Initially, we had issues with manual deployments and configuration inconsistencies. After implementing this structured CI/CD architecture with Infrastructure as Code, deployments became repeatable, traceable, and significantly reduced manual errors.
________________


🔹 2. Describe the CI/CD pipeline architecture you implemented. What stages did you create and why?
We implemented a multi-stage YAML pipeline.
CI Stages:
* Build

* Unit Test

* SonarQube Analysis

* Docker Build & Push

CD Stages:
   * Dev Deployment

   * QA Deployment

   * UAT Deployment

   * Production Deployment (with approval)

We separated CI and CD to ensure only validated artifacts move forward. Production had manual approval configured through Azure DevOps Environments to enforce governance.
Earlier, deployments were tightly coupled and risky. After implementing stage separation and approval gates, we improved release confidence and reduced failed production deployments.

🔹 3. What branching strategy did you use and why?
We used a modified GitFlow strategy.
Feature branches were created for development work. Developers raised PRs to merge into main. Direct commits to main were restricted through branch policies.
We chose this approach because multiple developers were working simultaneously. Feature branching helped isolate work and avoid conflicts.
Earlier, without proper branch policies, unstable code was getting merged. After enforcing PR validation and mandatory builds, code stability improved significantly.
________________

We used the same Docker image across all environments to maintain consistency.
Initially, configuration differences caused unexpected production issues. After isolating environment configurations and avoiding image rebuilds per environment, deployments became predictable and stable.

🔹 6. What was your rollback strategy during production failures?
Our primary rollback strategy was image version rollback.
Since Docker images were tagged with Build ID and stored in ACR, we could redeploy a previous stable version using Helm.
In one incident where a release caused memory issues, we quickly redeployed the previous image version. Downtime was avoided due to rolling updates.
This immutable artifact strategy made rollback fast and reliable.

🔹 7. Did you implement Blue-Green or Canary deployments? If yes, explain how?
We implemented rolling updates by default in Kubernetes. For critical releases, we simulated a blue-green approach using separate deployments and service switching.
Traffic was initially routed to the old version. After validating the new deployment in production namespace, service selectors were switched.
This reduced deployment risk and allowed safe production validation before full traffic cutover.

🔹 23. How did you manage Terraform state files?

                                       * Separate state file

Before this setup, we faced issues with overlapping changes when multiple engineers ran Terraform locally. Remote backend with locking eliminated conflicts and ensured infrastructure consistency.

What is a Terraform state file?

A state file is a record of what Terraform has created in your cloud environment. It keeps track of all resources, their IDs, properties, and relationships. Terraform uses this file to know what exists, what needs to be updated, and what should be destroyed when you apply changes.

Think of it like a map of your infrastructure that Terraform keeps to avoid creating duplicate resources or accidentally deleting something important.

Key Points

Tracks infrastructure – keeps the current status of VMs, networks, databases, etc.

Required for updates – Terraform compares the state file with your configuration to decide what to change.

Can be stored locally or remotely – in a file (local terraform.tfstate) or in a backend like Azure Storage, S3, etc.

Shared in teams securely – storing remotely allows multiple people to work on the same infrastructure safely.

Example for Interview

💬 “A Terraform state file is a JSON file that keeps track of all the resources Terraform manages. When we run terraform apply, Terraform looks at the state file to see what already exists, what needs to be added, and what can be updated or deleted. For team projects, we store the state file in a remote backend like Azure Storage to allow safe collaboration and prevent conflicts.”

## 🔹 Monitoring, Logging & Troubleshooting
29. How did you implement monitoring and logging after deployment (Azure Monitor, App Insights)?
After deploying applications to AKS, we enabled Azure Monitor for Containers and integrated the cluster with a Log Analytics Workspace. This allowed us to collect node-level and pod-level metrics such as CPU usage, memory consumption, restarts, and network traffic.
For application-level monitoring, we integrated Application Insights into the application code. This helped track request rates, response times, failure rates, dependency calls, and exceptions. Logs were centralized into Log Analytics for advanced querying using KQL.
We configured alert rules for:
                                                   * High CPU or memory utilization

                                                   * Pod crash loops

                                                   * Increased error rate

                                                   * Slow response times

Alerts were connected to email notifications and action groups.
This monitoring setup helped us proactively identify performance bottlenecks and reduced mean time to resolution (MTTR) during incidents.
________________


30. Describe a real-time production issue you faced. How did you troubleshoot and resolve it?
In one production deployment, users started experiencing intermittent 502 errors after a new release.
First, we checked Azure DevOps deployment logs to confirm the release was successful. Then we monitored AKS using Azure Monitor and noticed frequent pod restarts.
Using kubectl describe pod and container logs, we identified that the new version had insufficient memory limits configured. The application was getting OOMKilled under peak traffic.
To resolve it:
                                                      * We temporarily scaled replicas to handle traffic.

                                                      * Updated resource requests and limits in the Helm values file.

                                                      * Redeployed using rolling update to avoid downtime.

After deployment, we monitored metrics closely to ensure stability.
As a preventive action:
                                                         * We implemented proper load testing before production releases.

                                                         * Reviewed resource sizing strategy.

                                                         * Added alerts for memory pressure and pod restart thresholds.

This experience reinforced the importance of monitoring, proper resource planning, and validation before production rollout.

🔹 9. Terraform deployment fails due to state lock issues. How would you resolve it?

                                                                                                         1. Verify if another pipeline or user is actively running Terraform.

To prevent recurrence:
                                                                                                            * We restricted Terraform execution only through pipeline (not local machines).

                                                                                                            * Enabled proper pipeline timeout handling.

                                                                                                            * Avoided parallel apply operations.

This reduced state conflicts and ensured infrastructure consistency.

🔹 26. You need to build Docker images and push them to ACR automatically. How would you design it?
Answer :
So in my retail AKS project, we fully automated the Docker build and push process using Azure DevOps YAML pipelines.

After the image is successfully pushed to ACR, the deployment stage pulls that exact tagged version and deploys it to AKS using Helm.
This way:
                                                                                                                                                                                                                                    * No manual Docker builds

                                                                                                                                                                                                                                    * Full traceability (commit → build → image → deployment)

                                                                                                                                                                                                                                    * Secure and compliant pipeline

🔹 27. AKS works in Dev but crashes in Prod. How would you debug?

                                                                                                                                                                                                                                       * Secrets from Key Vault

                                                                                                                                                                                                                                       * Resource limits (CPU/memory)

                                                                                                                                                                                                                                       * HPA settings

                                                                                                                                                                                                                                       * Network policies or NSGs

In one real scenario, the issue was memory limits. Dev traffic was low so it worked fine, but in Prod under heavy load, pods were getting OOMKilled. We increased memory limits and fine-tuned HPA, and the issue was resolved.
I also use Azure Monitor and Log Analytics to analyze container logs and check for dependency failures or connection issues.
If it's a critical issue affecting customers, I immediately roll back to the previous stable build using Helm rollback or redeploying the previous artifact.
So my approach is:
Observe → Compare configs → Check logs → Validate resources → Rollback if needed.

🔹 28. You must implement rolling updates. How would you configure it?

🔹 31. Multiple pipelines are queued and waiting for agents. How would you optimize agent usage?

This reduced queue time significantly and improved overall delivery speed.
